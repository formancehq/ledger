# Antithesis chaos workload

This directory contains the workload that the [Antithesis](https://antithesis.com)
hypervisor drives against a multi-node `ledger-v3-poc` cluster under continuous
fault injection (node kills, network partitions, clog, disk pressure, â€¦).

## Layout

| Path | What lives here |
|---|---|
| `workload/` | Go module with all driver binaries (one binary per `bin/cmds/<template>/<driver>/main.go`). |
| `workload/internal/` | Shared helpers (gRPC client, error classification, ledger helpers, assertions wrappers, K8s client). |
| `workload/internal/block/` | Composable transaction scenarios used by `singleton_driver_scenario_blocks`. |
| `config/` | Docker-Compose-based local stack (3-node Raft + etcd + MinIO). **Legacy / local-iteration only.** |
| `k8s/` | Operator-managed StatefulSet deployment used by Antithesis itself. Manifests under `k8s/manifests/` are generated by `generate-manifests.sh`. |
| `run_model_test.sh` | Local-only runner for `singleton_driver_model`. The 60+ chaos drivers under `main/` only run on Antithesis. |
| `Justfile` | Top-level recipes (`just k8s-push-images`, `just compose-up`, â€¦). |

## How drivers are discovered

The workload `Dockerfile` compiles **every** `bin/cmds/<template>/<driver>/main.go`
in parallel (`find ... | xargs go build`) and copies the binaries to
`/opt/antithesis/test/v1/<template>/<driver>`. Antithesis scans this tree at
boot and picks **exactly one template** per execution history. Today we ship
two templates:

- `main/` â€” the broad fan-out of property drivers (~60 binaries, mostly `parallel_driver_*`).
- `model/` â€” the single `singleton_driver_model` conformance check.

Adding a driver is just adding a directory under `bin/cmds/main/` with a
`main.go`. No manifest to update; no Dockerfile to touch.

## Driver naming convention

The prefix encodes **how Antithesis schedules the binary**:

| Prefix | Scheduling | Use for |
|---|---|---|
| `parallel_driver_` | Many concurrent instances throughout the run. | Property checks that benefit from concurrent load (transactions, metadata, queries, â€¦). |
| `singleton_driver_` | Exactly one instance at a time, runs to completion. | Cluster-wide operations (scaling, rolling restart, backup, signing key rotation). |
| `eventually_` | Runs at the **end** of the workload, after writers have quiesced. | Cross-checks that only make sense after the system has had time to converge (balance audits, cross-node identity). |
| `first_` | Runs **before** the parallel pool starts. | One-shot setup (e.g. `first_default_ledger`). |

## Driver-side conventions

These conventions are what every driver in this tree should follow. They exist
because chaos testing is unforgiving: a driver that misclassifies an error or
swallows a stream truncation will silently report "all green" on a real bug.

### Error classification

The workload uses a layered predicate set (`internal/client.go`):

- `IsTransient(err)` â€” retry-safe set, what the retry interceptor handles.
  Covers `Unavailable | DeadlineExceeded | ReadIndexNotCaughtUp |
  ExternalServiceError`.
- `IsCanceled(err)` â€” local ctx is dead (driver shutting down). Not a
  finding; the driver just exits.
- `IsTolerated(err)` â€” `nil | IsTransient | IsCanceled`. **This is what
  Sometimes() probes use**: `assert.Sometimes(internal.IsTolerated(err),
  "should be able to X", details)`. Using `IsTransient` directly here would
  flip the per-driver Sometimes to "never true" when ctx cancellation
  dominates a chaotic run.
- `IsAmbiguousCommit(err)` â€” strict subset of `IsTransient` where the
  request may have committed despite the error (today: `DeadlineExceeded`).
  Drivers asserting on post-commit state can use this to decide whether to
  verify read-after-write even on the error branch.
- `IsClassified(err)` â€” `nil | IsTransient | IsCanceled | <business code>`
  (deliberately excludes `Aborted`). Business codes are `NotFound`,
  `AlreadyExists`, `InvalidArgument`, generic `FailedPrecondition`;
  drivers asserting on a specific business outcome check the precise
  reason via `internal.HasErrorReason(err, â€¦)` (e.g.
  `domain.ErrReasonInsufficientFunds`), never a coarse "is it business"
  predicate. A **classify interceptor** asserts `Always(IsClassified)`
  on every RPC return. An unclassified code (`Internal` panic,
  `ResourceExhausted` from a future rate-limiter, an unhandled gRPC codeâ€¦)
  surfaces as a global `Unreachable` with the method and code in Details â€”
  the workload's predicate set is incomplete and silent paths exist.

`Aborted` is deliberately NOT in `IsTransient`: the gRPC spec allows
business "please retry" semantics there, and we have no audited guarantee
the server keeps Aborted as a pure transport transient. Letting the
classify interceptor surface it (as "Always(IsClassified)" Details
`code=Aborted`) is intentional â€” if it shows up under chaos, that is a
finding worth triaging.

`IsUnavailable(err)` is deliberately narrow â€” it backs the gRPC service
config's retryable-codes list. Using it as a Sometimes tolerance predicate
masks `DeadlineExceeded` / `ReadIndexNotCaughtUp` / `ExternalServiceError`
and silently shorts the driver â€” exactly the bug the audit in
`refactor(antithesis): chaos error classification + workload cleanup`
caught across ~21 drivers.

For idempotent creates, the canonical pattern is:

```go
if err != nil {
    if internal.IsTransient(err) || internal.IsCanceled(err) {
        return // retry next round / driver shutting down
    }
    st, _ := status.FromError(err)
    if st.Code() != codes.AlreadyExists {
        assert.Unreachable("X creation returned unexpected error", details)
        return
    }
    // AlreadyExists: another worker beat us, fall through.
}
```

For the simpler "tolerate the error and skip" path the canonical pattern is:

```go
if !internal.IsTolerated(err) {
    assert.Unreachable("X returned unexpected error", details)
    return
}
if err != nil {
    return
}
```

### Lifecycle & context

- Parallel drivers should go through `internal.RunDriver(name, fn)` â€” it sets up
  the gRPC client, picks a random ledger, and bounds the run with the standard
  10 min deadline.
- Drivers that own their `main()` (singletons, first-, eventually-, or
  parallel drivers that need to create their own ledger from scratch) should
  use `internal.SingletonContext()` (90 min) or `internal.DriverContext()`
  (10 min) instead of bare `context.Background()`, so a SUT hang surfaces as a
  timed-out context.
- Driver-owned ledgers must use a typed `internal.OwnedLedgerPrefix`
  constant (e.g. `internal.PrefixInsufficientFunds.New()` or
  `.WithSeed(run)` when several names share a seed). The constant must
  be declared in `internal/ledger.go` and registered in
  `ownedLedgerPrefixes`; `TestOwnedLedgerPrefixes_NoOverlap` pins the
  no-shared-prefix invariant so a typo like `"lrec"` (which does not
  match `lrecreate-N`) is caught at test time, not in a chaos run.

### Transaction validation

When a driver creates a transaction and uses any field of the response,
prefer `internal.CheckCreatedTransaction(resp, details)` over the manual
`ExtractCreatedTransaction + CheckPostCommitVolumes` pair. It validates that
`balance == input - output` on every touched account for free.

### Antithesis SDK usage

- **Every `assert.Always` / `Sometimes` / `Reachable` / `Unreachable` MUST have a
  globally-unique name.** Antithesis indexes assertions by name; two sites
  sharing one name collapse into one signal and the triage UI shows you a
  single average instead of three failure modes.
- `Sometimes` is a coverage sonde â€” Antithesis prioritizes paths that make
  more `Sometimes` calls satisfied. Use it to mark expected outcomes (`err
  == nil || IsTransient(err)`) even when no invariant is at stake; it tells
  the fuzzer "this branch matters."
- A `Reachable("X")` with no upstream `Sometimes` that fires when X is true
  is passive: Antithesis cannot bias toward making X happen. Prefer pairing
  them when the path is fragile.
- Never use `panic` / `log.Fatal` / `os.Exit` to signal a finding from a
  driver â€” they kill the worker without producing a structured signal.
  `setup` programs (`first_default_ledger` etc.) may use `log.Fatalf` for
  client-construction failures, since they are infrastructure errors, not
  findings.
- Stream errors deserve classification, not blanket swallow: `if err != nil
  && !IsTransient(err) { assert.Unreachable(...) }` before skipping is the
  minimum bar. Otherwise an `InvalidArgument` on a `Recv()` is undistinguishable
  from a partition-induced `Aborted`.

### What *not* to do

- `time.Sleep` for convergence â€” it is non-deterministic and Antithesis cannot
  explore other timings. Loop with `assert.Sometimes(condition, â€¦)` instead,
  bounded by the driver's context deadline.
- Hand-roll a retry loop that classifies errors itself â€” let the gRPC
  interceptors in `internal/client.go` handle transients. Drivers only see
  errors that survived `retryMaxAttempts` retries.
- Swallow an error with `if err != nil { continue }` â€” at least log via
  `LogCleanupError` (for cleanup paths) or `assert.Reachable("X skipped due
  to error", â€¦)` so the path stays visible in the trace.

## Running locally

```sh
# Model-driver only (single Go binary against a 1-node local cluster).
./run_model_test.sh

# Full Compose stack (legacy, 3-node Raft + etcd + MinIO + NATS).
just compose-up
just compose-down

# Build and push the workload + ledger images to Antithesis's registry.
just k8s-push-images
```

`run_model_test.sh` only exercises `singleton_driver_model` â€” the 60+
property drivers under `main/` are tested exclusively by the Antithesis
hypervisor.

## Adding a new driver

1. Create `bin/cmds/main/<prefix>_<name>/main.go`.
2. Pick the right prefix (`parallel_driver_`, `singleton_driver_`,
   `eventually_`, `first_`).
3. Wire it through `RunDriver` when possible.
4. If the driver owns its ledgers, declare a `OwnedLedgerPrefix` constant
   in `internal/ledger.go` and append it to `ownedLedgerPrefixes`. Use
   `<Prefix>.New()` or `.WithSeed(seed)` in the driver â€” never
   `fmt.Sprintf("name-%d", â€¦)`.
5. Build locally: `cd workload && GOROOT= go build ./bin/cmds/main/...`.

No other files to edit â€” the Dockerfile picks up the new directory automatically.
